# 桥检AI（BridgeAI）安卓APP MVP研发PRD

## 1. 文档定位

本PRD用于统一桥检AI项目的MVP研发基线，作为产品、设计、安卓开发、算法、后端和测试的共同执行依据。

本文件明确采用以下基线：

1. 产品形态是 `安卓APP`，不是小程序。
2. MVP优先服务一线桥梁检测作业，不追求平台大而全。
3. 主流程采用 `桥梁/任务 -> 构件 -> 采集 -> AI识别 -> 人工校核 -> 报告`。
4. 以最新原型和后期文档认知为准，不再沿用早期“首页直接拍照即主流程”的旧思路。

---

## 2. 产品定义

### 2.1 一句话定义

桥检AI是一款面向桥梁检测员的安卓现场作业APP，用于完成桥梁构件级病害采集、AI辅助识别、人工校核、离线存储、报告生成和结果上报。

### 2.2 核心价值

1. 把桥梁检测从“照片散落、记录靠手写、报告靠整理”改成结构化数字作业。
2. 让每张照片、每条病害、每份报告都能明确归属到桥梁、项目和构件。
3. 在弱网或无网场景下仍能完成核心检测任务，回网后统一同步。
4. 用AI辅助识别提升效率，但保留人工校核作为工程结果的最终确认机制。

### 2.3 MVP目标

MVP只解决“现场作业闭环”：

1. 组长创建桥梁与检测任务。
2. 检测员按构件采集照片或视频。
3. APP完成本地或半本地AI识别。
4. 检测员人工修正病害信息。
5. 数据离线保存并在联网后同步。
6. 项目负责人汇总生成标准检测报告。

### 2.4 MVP不做

1. 不做小程序版本。
2. 不做无人机接入和实时视频流巡检。
3. 不做复杂审批流和多级组织架构。
4. 不做统计大屏和BI分析平台。
5. 不做高度自定义报告模板系统。
6. 不做与OA、财务、政务平台等第三方系统集成。
7. 不做iOS首发，iOS仅作为后续预研方向。

## 3. 用户与角色

### 3.1 目标用户

#### A. 检测员

负责现场执行检测任务，是MVP的主力用户。

核心诉求：

1. 任务清楚。
2. 操作简单。
3. 离线可用。
4. 采集结果不混乱。
5. 报告素材自动沉淀。

#### B. 组长/项目负责人

负责项目创建、检测计划、进度跟踪、报告确认。

核心诉求：

1. 能分配任务。
2. 能看到完成进度。
3. 能汇总多人检测数据。
4. 能生成并审核标准报告。

#### C. 管理员

负责基础账号和桥梁档案维护。MVP阶段权限保持简化。

### 3.2 角色权限

| 角色 | 核心权限 | 限制 |
| --- | --- | --- |
| 检测员 | 查看任务、进入检测、拍照录像、AI识别、人工修正、保存本地、同步、查看自己参与的报告 | 不可删除桥梁，不可确认最终报告 |
| 组长/项目负责人 | 拥有检测员全部能力，且可创建项目、制定计划、分配任务、生成报告、确认报告 | 不做复杂组织级管理 |
| 管理员 | 维护桥梁档案、账号与基础配置 | 不引入复杂RBAC |

## 4. 产品原则

1. `构件优先`：先明确检测对象，再进行采集。
2. `离线优先`：核心作业在无网环境也要可完成。
3. `安卓现场优先`：适配单手、戴手套、强光、弱网、长时使用。
4. `AI辅助而非替代`：AI给出建议，人工做最终确认。
5. `报告导向`：所有检测数据都要服务最终报告生成。
6. `先闭环后扩展`：先打通现场闭环，再考虑无人机、iOS、大屏等扩展。

### 5.1 底部导航

MVP建议采用以下4个一级Tab：

1. `任务`
2. `桥梁`
3. `报告`
4. `我的`

说明：

1. 不再把“AI检测首页”作为唯一主入口。
2. 真正的检测入口应从任务或桥梁进入，再下钻到构件检测页面。
3. 相机能力只在具体构件场景下唤起，避免照片失去归属。

## 6. 端到端业务流程

### 6.1 标准流程

1. 组长创建桥梁档案或选择已有桥梁。
2. 组长创建检测项目。
3. 组长制定检测计划，拆分到构件级。
4. 组长分配检测员。
5. 检测员进入任务，逐个构件采集照片或视频。
6. APP触发AI识别。
7. 检测员修正病害类型、等级、位置和备注。
8. 数据保存到本地数据库。
9. 联网后同步到服务端。
10. 组长确认项目数据完整。
11. 系统自动生成报告草稿。
12. 组长审核并确认报告。
13. 导出PDF或执行上报。

### 6.2 关键流程约束

1. 每条检测数据必须关联到 `bridge_id`、`project_id`、`component_type`、`component_number`。
2. 每张素材必须有明确归属，禁止出现“无构件上下文的自由拍照记录”进入正式报告。
3. AI结果可以被人工修改，人工结果优先级高于AI结果。
4. 报告中的检测数据和原始照片不可被随意篡改，只允许补充备注、建议和签字信息。

## 7. 核心功能需求

## 7.1 模块A：任务与项目

### 7.1.1 任务列表

功能目标：让检测员一打开APP就知道“今天做什么”。

功能要求：

1. 展示我参与的项目列表。
2. 支持按 `待开始/进行中/已完成` 筛选。
3. 每个项目卡片展示桥梁名、项目类型、时间范围、负责人、完成进度。
4. 点击项目进入检测页。

验收标准：

1. 列表加载时间不超过1秒。
2. 检测员能在3步内进入具体构件检测页。

### 7.1.2 项目检测页

功能目标：作为现场检测的主控台。

功能要求：

1. 展示桥梁基础信息。
2. 展示构件清单，按 `上部结构/下部结构/附属设施` 分组。
3. 每个构件显示 `未检测/已采集/已识别/已完成` 状态。
4. 显示项目整体进度和构件完成数。
5. 对存在未识别素材的构件显示 `批量AI识别` 入口。
6. 当所有必检构件完成后显示 `生成报告` 入口给组长。

验收标准：

1. 构件状态更新及时。
2. 进度统计准确。
3. 可清晰看出哪些构件还未完成。

## 7.2 模块B：桥梁档案

### 7.2.1 桥梁列表

功能要求：

1. 支持搜索桥梁名称和编码。
2. 支持按桥梁类型筛选。
3. 展示结构类型、建成年份、最近检测状态。
4. 支持从桥梁详情直接开始检测。

### 7.2.2 桥梁详情

功能要求：

1. 展示桥梁基础信息、位置信息、尺寸参数、建成年份、养护单位。
2. 展示历史检测记录和关联报告概览。
3. 提供 `选择此桥梁并开始检测` 按钮。

### 7.2.3 桥梁新增与编辑

MVP字段：

1. 桥梁名称
2. 桥梁编码
3. 所属路线
4. 桥梁类型
5. 结构类型
6. 建成年份
7. 全长
8. 桥宽
9. 主跨
10. 地址描述
11. 养护单位
12. 备注

## 7.3 模块C：检测计划

### 7.3.1 项目创建

功能要求：

1. 选择桥梁。
2. 选择项目类型：定期检测、特殊检测、应急检测。
3. 设置开始和结束时间。
4. 指定项目负责人。

### 7.3.2 构件计划

功能要求：

1. 支持预置构件模板。
2. 允许负责人新增、删除、调整构件项。
3. 支持为构件分配检测员。
4. 支持为构件配置关注病害类型。

MVP建议的构件模板：

1. 上部结构：主梁、横梁、桥面板、伸缩缝、支座。
2. 下部结构：桥墩、桥台、墩柱、承台、基础。
3. 附属设施：栏杆、护栏、排水系统、标志标线、防撞护栏。

## 7.4 模块D：构件采集

### 7.4.1 构件采集页

功能要求：

1. 显示当前桥梁、构件类型和构件编号。
2. 支持拍照。
3. 支持录像。
4. 支持从相册导入图片或视频。
5. 支持预览和删除已采素材。
6. 保存后自动归档到当前构件。

交互要求：

1. 拍照按钮足够大，适合现场快速操作。
2. 支持后置摄像头。
3. 支持断点返回，不丢失已拍数据。

验收标准：

1. 素材保存成功率高于99%。
2. 单个构件可累计保存多张照片和多个视频。
3. 不允许素材挂错构件。

### 7.4.2 采集规则

1. 所有采集动作都必须发生在具体构件页面中。
2. 同一构件可重复进入补拍。
3. 删除素材应有确认提示。
4. 若采集完成但未识别，构件状态记为 `已采集`。

## 7.5 模块E：AI识别

### 7.5.1 MVP阶段AI策略

MVP支持两种运行模式：

1. `模式1：模拟识别`
2. `模式2：真实识别接口占位`

目的：

1. 先打通产品闭环和数据结构。
2. 再平滑接入真实YOLO/TFLite模型。

### 7.5.2 支持病害类型

MVP统一使用以下8类，后续以算法标签体系为准维护：

1. 裂缝
2. 混凝土剥落/露筋
3. 锈蚀
4. 混凝土空洞
5. 表面劣化
6. 渗水/潮湿
7. 收缩裂缝
8. 施工缺陷

### 7.5.3 识别结果内容

每条AI结果至少包含：

1. 病害类型
2. 置信度
3. 缺陷框坐标
4. 严重等级建议
5. 识别时间
6. 模型版本

### 7.5.4 识别触发方式

MVP采用：

1. 单构件识别
2. 项目页批量识别

规则：

1. 已采集未识别的构件可统一批量处理。
2. 识别完成后构件状态变为 `已识别`。
3. 识别失败时保留原素材并允许重试。

## 7.6 模块F：人工校核

### 7.6.1 构件结果页

功能要求：

1. 查看本构件所有照片与视频。
2. 查看AI识别结果。
3. 支持编辑病害类型。
4. 支持编辑病害等级。
5. 支持编辑位置描述。
6. 支持填写备注。
7. 支持重新拍摄和补拍。

### 7.6.2 编辑规则

1. AI结果只是建议值。
2. 人工修改后写入正式检测数据。
3. 所有编辑要记录更新时间。
4. 后续服务端可扩展修改日志，MVP本地先保留最近更新时间字段。

### 7.6.3 构件完成规则

当满足以下条件时构件可标记完成：

1. 已有素材。
2. AI已识别或被人工确认无需识别。
3. 至少存在一条有效病害记录或明确标注“未发现病害”。

## 7.7 模块G：离线存储与同步

### 7.7.1 离线能力

MVP必须支持：

1. 离线查看本地任务数据。
2. 离线采集素材。
3. 离线保存AI结果和人工编辑结果。
4. 离线生成待同步队列。

### 7.7.2 数据同步

功能要求：

1. 在“我的-数据同步”查看未同步记录数。
2. 支持手动同步。
3. 展示同步进度。
4. 同步成功后更新本地状态。
5. 同步失败时保留数据并允许重试。

同步范围：

1. 桥梁基础数据更新
2. 项目与计划变更
3. 检测记录
4. 素材文件
5. 报告草稿

### 7.7.3 冲突处理原则

MVP简化规则：

1. 人工最终编辑结果优先。
2. 本地未同步数据优先保留，不自动覆盖。
3. 同步冲突先提示，再由负责人确认。

## 7.8 模块H：报告中心

### 7.8.1 报告列表

功能要求：

1. 按项目展示报告。
2. 支持搜索报告名和桥梁名。
3. 展示状态：草稿、已完成、已上报。
4. 支持进入报告预览页。

### 7.8.2 报告生成

触发条件：

1. 项目所有必检构件已完成。
2. 相关数据已同步或已在本地汇总完成。
3. 由组长主动触发生成。

报告结构：

1. 项目概况
2. 桥梁基本信息
3. 检测概况
4. 病害清单
5. 技术状况评定
6. 养护建议
7. 附件照片
8. 签字栏

### 7.8.3 报告预览与编辑

可编辑内容：

1. 病害备注
2. 养护建议
3. 签字信息

不可编辑内容：

1. 原始检测照片
2. 检测记录主体数据
3. AI原始输出快照

### 7.8.4 导出与上报

MVP要求：

1. 支持导出PDF。
2. PDF文件命名规范固定。
3. 支持标记为已上报。
4. 分享能力先保留系统级分享入口，不做复杂链接体系。

## 7.9 模块I：我的

功能要求：

1. 显示当前用户信息。
2. 展示离线模式开关。
3. 展示未同步数据数。
4. 进入同步页。
5. 提供帮助中心、关于我们、退出登录。

## 9. 非功能要求

### 9.1 性能

1. 冷启动不超过3秒。
2. 普通页面切换不超过500ms。
3. 构件列表100条以内可流畅滚动。
4. 单次本地识别目标时间不超过2秒。

### 9.2 稳定性

1. 连续使用4小时不崩溃。
2. 采集过程异常退出后，已保存素材不丢失。
3. 离线状态下不能因网络异常导致页面阻塞。

### 9.3 安全

1. 账号信息本地安全存储。
2. 本地敏感数据应具备加密能力预留。
3. 同步走HTTPS。
4. 报告生成后保留状态流转记录。

### 9.4 现场适配

1. 主要按钮不小于48dp。
2. 适配强光下可读性。
3. 重要信息采用高对比度显示。
4. 主要流程支持单手和快速点按。

### 9.5 安卓设备范围

1. 仅做安卓APP。
2. 优先适配 Android 10 及以上。
3. 优先适配 6.5 英寸到 10 英寸手机和平板。

## 10. 技术与实现约束

本节只定义研发约束，不替代详细技术方案。

1. 客户端为安卓APP，建议原生Kotlin实现。
2. 采集能力需要调用系统摄像头，支持拍照和录像。
3. 本地数据库需支持离线可用。
4. AI模块需兼容“本地模拟”和“真实模型/接口接入”两种模式。
5. 同步能力需支持失败重试和断点继续。
6. PDF导出必须在安卓端可执行。

---

## 11. MVP验收标准

### 11.1 功能验收

1. 组长可创建桥梁、项目和构件计划。
2. 检测员可按构件完成拍照或录像采集。
3. 每条检测数据都可追溯到桥梁、项目和构件。
4. AI识别链路可运行，至少支持模拟模式。
5. 检测员可在构件结果页完成人工校核。
6. 离线保存和手动同步可用。
7. 系统可生成报告草稿并导出PDF。

### 11.2 业务验收

1. 单个桥梁项目可完整跑通一次检测闭环。
2. 多名检测员的数据可汇总到同一项目。
3. 报告内容能支撑负责人完成一次对外交付。

### 11.3 体验验收

1. 现场检测主流程不超过3层页面跳转。
2. 用户不会出现“拍了照片但不知道归属到哪里”的情况。
3. 用户可以明确知道哪些数据未同步。

## 12. 研发交付建议

为了让MVP真正可落地，建议交付拆成4个最小版本：

1. `V0.1 原型闭环`：桥梁、构件、采集、本地保存。
2. `V0.2 识别闭环`：AI模拟、人工校核、状态流转。
3. `V0.3 协同闭环`：项目、计划、任务、同步。
4. `V1.0 交付闭环`：报告生成、PDF导出、稳定性优化。

---

## 13. 本PRD的最终判断

桥检AIMVP的本质不是“图片识别工具”，而是“安卓端桥梁检测作业系统”。

项目成功的关键不在于首页多炫，而在于以下四件事是否扎实：

1. 构件归属是否清晰。
2. 离线作业是否可靠。
3. 数据汇总是否稳定。
4. 报告输出是否可交付。
